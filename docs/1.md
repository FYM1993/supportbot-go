## 📋 一、项目介绍（1分钟版 · Agent 视角）

这是一个面向**智能客服场景的 AI Agent 系统**，用 Golang 实现，具备分类路由、RAG 知识库、Function Calling 工具调用等完整 Agent 能力链路。

**核心 Agent 能力：**
- **Agent  orchestration（编排）**：LLM 分类器负责意图识别，将用户请求路由到对应专业 Agent
- **RAG 知识检索 Agent**：Embedding + 向量检索 + LLM 生成，解决知识型问答与幻觉问题
- **Tool-Use Agent（assistant）**：支持 Function Calling，可调用订单、商品、物流等业务工具
- **实时通信层**：WebSocket 双向通道，支持 Agent 异步回调、流式返回

**技术栈：** Golang + WebSocket + Redis + LLM(通义千问) + RAG + 向量检索 + Function Calling

**项目地址：** https://github.com/FYM1993/supportbot-go

### 2.2 与字节面试标准的 Agent 架构对比

| 维度 | 本项目实现 | 工业级 Agent 架构（面试可扩展讲） |
|------|-----------|----------------------------------|
| **编排方式** | 分类器路由，按类别派发到单一 Agent | 可扩展为 ReAct+ToT 规划、多步拆解、反思校验 |
| **工具调度** | Function Calling 循环，LLM 决策调用 | 规则过滤 + 语义路由 + 后置校验三级调度 |

| **框架** | 自实现（Golang） | LangGraph/LangChain 等（可选） |

**面试话术：** 本项目是面向智能客服的轻量级 Agent 架构，采用分类路由 + 多专业 Agent 模式。与完整 ReAct 的区别是：我们通过「分类器」做粗粒度路由，而不是在单 Agent 内做多步规划。这样设计是为了客服场景的「快速响应」和「职责清晰」，每个 Agent 专注一类任务。若要做复杂多步任务，可在分类器后增加「规划模块」，引入 ReAct/ToT。

## 🧠 三、Agent 核心能力深度拆解

### 3.1 ReAct 与 Tool-Use 循环（assistant 实现）

**本项目实现的是「简化版 ReAct」：**

```
用户问题 → LLM + Tools → 是否调用工具？
    ├─ 是：执行工具 → 结果写回 messages → 继续下一轮 LLM
    └─ 否：生成最终回答 → 结束
```

**与标准 ReAct 的关系：**
- ReAct = **Reasoning**（推理）+ **Acting**（行动），显式 Thought-Action-Observation 循环
- 本项目：隐式实现「LLM 决策 → 工具执行 → 结果回传」的 Tool-Use 循环，等价于 ReAct 的 Action-Observation 部分
- 差异：没有显式「Thought」步骤，但通义千问的 Function Calling 内部会做推理和工具选择

**核心代码逻辑（面试可讲）：**

```go
// cmd/assistant/main.go 核心循环
for i := 0; i < maxIterations; i++ {
    resp, err := llmClient.ChatWithTools(messages, toolDefs)
    // 检查是否有工具调用
    if len(resp.Output.Choices[0].Message.ToolCalls) > 0 {
        messages = append(messages, assistantMsg)
        for _, toolCall := range assistantMsg.ToolCalls {
            result, err := toolRegistry.Execute(tc)
            messages = append(messages, toolResultMsg)
        }
        continue  // 继续下一轮，让 LLM 基于工具结果生成
    }
    // 无工具调用 → 最终回答
    finalResponse = resp.Output.Text
    break
}
```

**面试考点：**
- 为什么需要循环？因为一次可能调用多个工具，或需要多轮工具调用
- `maxIterations` 的作用？防止无限循环（类似字节面经提到的「总执行轮数上限」）
- 工具失败怎么处理？将 error 写入 tool 消息，让 LLM 基于错误信息重试或给出友好回复

### 3.2 分类器：Agent 路由编排

**职责：** 不是「多步规划」，而是**多 Agent 路由**，把用户请求分发给最合适的 Agent。

**流程：**
```
用户消息 → Redis 取最近 5 条历史 → 构建 Prompt → LLM 分类 
→ 保存历史 → 异步 HTTP 调用目标 Agent → Agent 处理完回调 im-demo 推送
```

**为什么取 5 条历史？**
- Token 限制、成本、相关性：更早对话与当前问题关联弱
- Redis List：LPUSH + LRANGE -5 -1，24h 过期

**分类配置化：**
```go
categories map[string]CategoryInfo  // category -> Agent URL
// 新增分类只需加配置，无需改核心逻辑
```

### 3.3 RAG Agent 全流程（必背）

**完整 RAG 链路（贴合字节面试「全链路讲清楚」要求）：**

#### 离线阶段（本项目简化版）
1. 文档加载与解析（支持多种格式）
2. 分块（Chunking）：固定长度 / 语义分块，可重叠
3. Embedding：通义 text-embedding-v2 → 1536 维向量
4. 存储：向量 + 原文 → 向量库（本项目为内存）

#### 在线阶段
1. Query 预处理：纠错、意图解析（可选）
2. Query Embedding：同一模型
3. 向量检索：余弦相似度 Top-K（本项目 K=3）
4. 上下文构建：`参考以下知识：[知识1][知识2][知识3]`
5. Prompt 注入：System + 检索上下文 + 用户问题
6. LLM 生成：仅基于检索内容回答，降低幻觉

#### 工业级 RAG 扩展（面试加分）
- **混合检索**：向量 + 关键词（ES）
- **Rerank**：二次排序模型提升精准度
- **效果评估**：召回率、精确率、幻觉率、LLM-as-Judge
- **分块策略**：语义分块、重叠、结构化分块（表格/代码）

**余弦相似度公式（可白板）：**
```
similarity = (A·B) / (||A|| × ||B||)  ∈ [-1, 1]
```

### 3.4 MCP、A2A（字节面试高频）

#### MCP（Model Context Protocol）

**定位：** Agent 如何**标准化接入**外部数据源、工具、系统，动态获取上下文。

**核心作用：**
- 统一协议：不同数据源/工具用同一套接口
- 可插拔：新增数据源无需改 Agent 核心逻辑
- 安全：权限、审计、访问控制

**RAG + MCP 结合价值：**
- 传统 RAG：每加一个数据源都要定制开发
- MCP：把 RAG 封装成 MCP Tool/Resource，Agent 通过协议调用
- 支持静态知识库 + 动态数据源（飞书、DB 等）实时接入

**本项目：** 未接入 MCP，但 Function Calling 与 MCP 的 tool 能力类似，可扩展为 MCP 服务端。

#### A2A vs MCP

| 维度 | A2A（Agent-to-Agent） | MCP（Model Context Protocol） |
|------|----------------------|------------------------------|
| 解决的问题 | Agent 之间如何协同 | Agent 如何拿到数据/工具 |
| 协议内容 | 任务分发、信息同步、结果传递 | 数据源、工具、Prompt 标准化接入 |
| 典型场景 | 多 Agent 协作完成任务 | RAG、数据库、API 对接 |

**一句话：** A2A 管「Agent 和 Agent 怎么一起干活」，MCP 管「Agent 怎么拿到干活需要的数据和工具」。

### 3.5 长上下文与记忆（面试必答）

**问题：** 对话轮数多、工具返回长 → 超出上下文窗口 → 遗忘目标、幻觉。

**方案对比：**

| 策略 | 做法 | 优点 | 缺点 |
|------|------|------|------|
| 截断 | 只保留最近 N 轮 | 简单 | 丢信息 |
| 摘要压缩 | 每轮用 LLM 做摘要，历史摘要+最近几轮 | 压缩比高 | 有损、延迟 |
| 记忆检索 | 历史 embedding 存向量库，按需召回 | 不受轮数限制 | 需额外存储 |
| 核心目标置顶 | 初始需求固定在 system prompt 最前 | 无损、防遗忘 | 占用 token |
| KV 缓存 | 推理时复用之前轮次 KV | 降延迟、省显存 | 依赖模型支持 |

**本项目：** Redis 只存最近 5 条，属于「截断 + 对话历史」的轻量方案。面试时可说：生产会考虑「记忆检索 + 摘要压缩」组合。

## 🔧 四、Agent 工程化与落地

### 4.1 项目背景与业务价值（字节必问）

**满分回答框架：**

```
背景：智能客服场景下，用户问题类型多样（订单、商品、知识库、闲聊），
需要不同专业能力处理。通用单模型无法兼顾准确性和效率，且无法对接内部系统。

痛点：
1. 人工客服重复处理同类问题，效率低
2. 知识分散，检索成本高，易遗漏
3. 订单/商品查询需要对接内部 API，通用 Chat 做不到

方案：做垂直领域的多 Agent 系统——
- 分类器：快速识别意图，路由到专业 Agent
- 订单/商品类 → assistant（Function Calling）
- 知识类 → knowledge-rag（RAG）
- 闲聊 → general-chat（纯 LLM）

效果预期：覆盖高频场景，任务成功率 >85%，响应时延可控，
为后续接入真实业务系统打基础。
```

---

### 4.2 研发中遇到的困难与解决（字节必问）

**示例回答（结合本项目可讲）：**

**困难 1：工具调用格式错误**
- 现象：LLM 返回的 JSON 格式不规范、参数缺失
- 解决：严格 JSON Schema 定义、prompt 中加入工具示例、工具执行失败把 error 回传 LLM 让其修正

**困难 2：分类不准导致路由错误**
- 现象：部分问题被分到错误 Agent
- 解决：优化 System Prompt（角色、任务、输出约束）、补充 Few-Shot、利用对话历史提升上下文理解

**困难 3：长对话遗忘初始问题**
- 现象：多轮后模型偏离主题
- 解决：限制历史条数（5 条）、在 system prompt 中强调当前任务目标

### 4.3 Agent 开发框架选型（字节必问）

**面试官常见问法：** 用的 LangGraph、LangChain 还是自研？

**本项目：** Golang 自实现，未用 LangChain/LangGraph。

**回答策略：**

```
选型：用 Golang 自实现了 Agent 核心逻辑，没有用 LangChain/LangChain。

原因：
1. 技术栈统一：整体是 Go 微服务，引入 Python 链式框架会增加复杂度和部署成本
2. 场景简单：客服场景以单步/少量步为主，分类路由即可，暂不需要复杂 ReAct 规划
3. 学习目的：自实现能更好理解 Tool-Use 循环、消息格式、工具注册等底层机制

若做复杂多步规划，会考虑：
- LangGraph：基于状态机的图结构，适合循环、分支、重试，比 Chain 更灵活
- 或自研 Golang 版「规划-执行-校验」状态机，保持技术栈一致
```

### 4.4 Agent 预期与当前短板（字节必问）

**满分回答框架：**

```
基础预期（已达成）：
- 能稳定完成分类路由、RAG 检索、工具调用

```

## 📚 五、多模态、微调、RAG 扩展（大厂基础题）

### 5.1 多模态大模型结构（简要）

- **模态编码器**：图像用 ViT/CLIP，音频用 Whisper 等
- **投影层**：编码器输出 → LLM 词嵌入空间对齐（线性/MLP/Q-Former）
- **LLM 解码器**：接收对齐后的多模态特征 + 文本，完成理解与生成

**架构类型：** CLIP 双塔、编码器-解码器融合（LLaVA/Qwen-VL）、统一语义空间（GPT-4V）

---

### 5.2 LoRA / QLoRA（简要）

**LoRA：** 低秩适配，冻结主模型，加 A(d×r)、B(r×d) 小矩阵微调，参数量极小。

**QLoRA 显存优化：**
1. 4 位 NF4 量化
2. 双量化（缩放常数再量化）
3. 分页优化器（CPU/GPU 分页）
4. 梯度检查点、梯度累积

---

### 5.3 多模态知识检索

**方案 1：转文本再检索**  
图/音 → 文本描述 → 文本 Embedding → 向量检索（本项目可用）

**方案 2：跨模态统一嵌入**  
CLIP/BLIP 等多模态模型，图/文映射到同一向量空间，支持「以图搜图」「以文搜图」

### Go 与并发（支撑层技术细节）

14. **双向索引、两把锁、心跳检测**  
    - 双向索引：`userID→Session` 和 `sessionID→userID`，O(1) 双向查找，支持重复登录踢旧连接  
    - 两把锁：Service.mu 保 map，Session.mu 保 WebSocket 写，细粒度锁实现用户级并发  
    - 心跳：30s 检测 + 60s 超时，容忍一次丢包，常驻 Goroutine 主动巡检

15. **Go map 并发不安全？为什么不用 sync.Map？**  
    - map 底层哈希表，并发写会改 buckets、count 等，扩容时更危险  
    - 读多写少 + 热点 key + 需原子更新两个 map，sync.Map 无法保证两 map 一致，RWMutex 更合适
